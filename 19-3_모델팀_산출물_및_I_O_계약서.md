# 19-3. 모델팀 산출물 및 I/O 계약서

기준 문서: [19-2](19-2_최종_백엔드_설계도.md), [05-6-4](05-6-4_프로그램_구조도_및_함수_변수_비전_명세서.md), [17-6-5](17-6-5_5m_models7_호환성정합_및_개선_계획서.md)

본 문서는 모델 담당자가 각자 학습한 모델을 백엔드·DB·대시보드와 연결하기 위해 **어떤 입력 파일을 쓰고, 어떤 코드 위치에서 구현하고, 어떤 산출물을 어떤 자료형으로 저장해야 하는지**를 명시한다.

`19-2`의 FastAPI 전환은 백엔드 내부 구현 변경이다. 모델 담당자의 입력 데이터, 학습 산출물, 제출 payload 계약은 유지한다.

---

## 1. 핵심 결론

| 구분 | 결정 |
| --- | --- |
| 운영 예측 대상 | `churn` 단일 라벨 |
| 예측 의미 | 향후 7일 이탈 확률 |
| 모델 입력 정본 | `data/processed/churn/models7/` |
| 현재 참조 원본 | `sample_project/data/processed_5m/models7/` |
| 모델 구현 코드 | `src/train_ml.py`, `src/train_dl.py`, 필요 시 `src/models/churn/` |
| 학습 노트북 | `notebooks/model_training/churn/` |
| 학습된 모델 파일 | `models/churn/{model_key}/` |
| 평가 산출물 | `data/processed/evaluation/churn/{model_key}/` |
| 백엔드 제출 단위 | `model_run_manifest.json` + 평가/모델 파일 경로 |

모델 담당자는 DB에 직접 쓰지 않는다. 모델 담당자는 파일 산출물과 manifest를 만들고, 백엔드가 `POST /models/submit`으로 등록한다.

## 3. 입력 파일 스키마

### 3.1 Tabular parquet 스키마

대상 파일:
- `CatBoost_train.parquet`, `CatBoost_test.parquet`
- `LightGBM_train.parquet`, `LightGBM_test.parquet`
- `XGBoost_train.parquet`, `XGBoost_test.parquet`
- `RandomForest_train.parquet`, `RandomForest_test.parquet`
- `DecisionTree_train.parquet`, `DecisionTree_test.parquet`
- `LogReg_train.parquet`, `LogReg_test.parquet`

필수 컬럼과 자료형:

| 컬럼 | dtype | 역할 | 비고 |
| --- | --- | --- | --- |
| `recency_days` | float64 | feature | 마지막 활동 후 경과일 |
| `tenure_days` | float64 | feature | 관측 기간 내 유저 존속 기간 |
| `ndays` | float64 | feature | 활동일 수 |
| `n_events` | float64 | feature | 전체 이벤트 수 |
| `n_view` | float64 | feature | view 이벤트 수 |
| `n_cart` | float64 | feature | cart 이벤트 수 |
| `n_remove_from_cart` | float64 | feature | remove_from_cart 이벤트 수 |
| `n_purchase` | float64 | feature | purchase 이벤트 수 |
| `avg_price` | float64 | feature | 평균 가격 |
| `purch_amt` | float64 | feature | 구매 금액 합계 |
| `churn` | int32 | target | 0 또는 1 |
| `user_id` | int64 | 식별자 | feature로 학습 금지 |

feature 순서는 반드시 아래 순서를 따른다.

```text
recency_days
tenure_days
ndays
n_events
n_view
n_cart
n_remove_from_cart
n_purchase
avg_price
purch_amt
```

`user_id`는 예측 결과를 다시 유저에게 붙이기 위한 식별자다. 모델 입력 feature에는 포함하지 않는다.

### 3.2 Canonical tabular 참고 스키마

정본 원본에 가까운 파일은 `train_tabular.parquet`, `test_tabular.parquet`이다. 이 파일은 아래 컬럼을 포함할 수 있다.

| 컬럼 | dtype | 용도 |
| --- | --- | --- |
| `user_id` | int64 | 식별자 |
| 10개 feature | int64/float32/float64 | 모델별 최적화 전 원본 feature |
| `churn` | int32 | 운영 라벨 |
| `churn_no_purchase` | int32 | 분석·비상용 라벨, 운영 학습에는 사용하지 않음 |

운영 모델은 `churn`만 사용한다. `churn_no_purchase`를 목표값으로 학습한 결과물은 `model_registry`의 active 운영 모델로 등록하지 않는다.

### 3.3 Sequence NPZ 스키마

대상 파일:
- `Transformer_train_seq.npz`
- `Transformer_test_seq.npz`

필수 key와 자료형:

| key | shape | dtype | 역할 |
| --- | --- | --- | --- |
| `X` | `(N, 17, 3)` | float32 | 유저별 시계열 feature |
| `churn` | `(N,)` | int32 | target, 0 또는 1 |
| `user_id` | `(N,)` | int64 | 식별자 |

`X`의 두 번째 축은 sequence length, 세 번째 축은 시점별 feature 수다. `train_seq_DL_opt.npz`처럼 `(N, 4, 3)` 형태를 쓰는 실험은 가능하지만, 운영 제출 시 manifest의 `preprocessing_config.seq_len`에 반드시 명시하고 Transformer 담당자와 백엔드 담당자가 같은 값을 사용해야 한다.

## 4. 코드와 노트북 저장 위치

### 4.1 운영 학습 코드

기본 진입점:

```text
src/train_ml.py
src/train_dl.py
src/compare.py
src/predict.py
```

모델별 코드가 길어질 경우 아래처럼 분리한다.

```text
src/models/churn/
├── catboost_trainer.py
├── lightgbm_trainer.py
├── xgboost_trainer.py
├── randomforest_trainer.py
├── decisiontree_trainer.py
├── logreg_trainer.py
└── transformer_trainer.py
```

규칙:
- 공통 CLI는 `src/train_ml.py`, `src/train_dl.py`에서 제공한다.
- 모델별 학습 함수는 `src/models/churn/*_trainer.py`에 둔다.
- 전처리 파일을 새로 만들거나 컬럼 순서를 바꾸는 코드는 모델 학습 코드에 넣지 않는다.
- 모델 학습 코드는 입력 파일 경로를 인자로 받고, 산출물 경로를 명시적으로 저장한다.

### 4.2 학습 노트북

노트북 저장 위치:

```text
notebooks/model_training/churn/
├── catboost_train.ipynb
├── lightgbm_train.ipynb
├── xgboost_train.ipynb
├── randomforest_train.ipynb
├── decisiontree_train.ipynb
├── logreg_train.ipynb
```

Colab에서 돌린 경우:

```text
notebooks/colab/churn/
├── catboost_train_colab.ipynb

```

노트북은 다음을 포함해야 한다.
- 사용 입력 파일 경로
- 학습 파라미터
- 최종 metric
- artifact 저장 경로
- `model_run_manifest.json` 생성 셀 또는 생성 스크립트 호출 셀

### 6.2 eval_predictions.parquet

테스트셋 예측 결과다. 대시보드의 score distribution, threshold, lift, calibration 재계산에 쓰인다.

| 컬럼 | dtype | 필수 | 설명 |
| --- | --- | --- | --- |
| `user_id` | int64 또는 string | Y | 유저 식별자 |
| `y_true` | int32 | Y | 실제 churn 라벨 |
| `y_score` | float64 | Y | churn 확률, 0.0~1.0 |
| `y_pred` | int32 | Y | threshold 적용 예측값 |
| `split` | string | Y | 보통 `test` |
| `threshold` | float64 | Y | y_pred 계산에 사용한 threshold |
| `model_name` | string | Y | 모델명 |

`y_score`는 반드시 positive class, 즉 `churn=1`의 확률이다.

## 7. 모델별 preprocessing_config 기준

| 모델 | input_format | scale | feature_order | 추가 설정 |
| --- | --- | --- | --- | --- |
| CatBoost | `parquet` | `none` 권장 | 10개 feature 고정 | `cat_features: []` |
| LightGBM | `parquet` | `none` 권장 | 10개 feature 고정 | `objective: binary` |
| XGBoost | `parquet` | `none` 권장 | 10개 feature 고정 | `objective: binary:logistic` |
| RandomForest | `parquet` | 파일 생성 기준 기록 | 10개 feature 고정 | `n_estimators` 기록 |
| DecisionTree | `parquet` | 파일 생성 기준 기록 | 10개 feature 고정 | `max_depth` 기록 |
| LogReg | `parquet` | `standard` 또는 파일 생성 기준 기록 | 10개 feature 고정 | scaler 사용 시 `preprocessor.joblib` 저장 |
| Transformer | `npz` | sequence 파일 기준 | `X` 내부 순서 | `seq_len`, `n_features` 필수 |

중요:
- 실제 입력 파일이 이미 scaled라면 `scale`에 `pre_scaled` 또는 `standard`를 명시한다.
- 학습 코드가 자체 scaling을 수행했다면 scaler 파일을 `models/churn/{model_key}/preprocessor.joblib`로 저장한다.
- 백엔드는 `preprocessing_config`를 보고 실시간 feature를 같은 방식으로 맞춘다.

## 12. 최종 확정 ML 시각화 산출물 기준

2026-06-22 기준 발표/대시보드에 올릴 ML 시각화 자료는 아래 7종으로 확정한다. 기존 15개 후보 전체를 모두 필수로 만들 필요는 없고, 모델팀은 아래 파일을 우선 산출한다.

### 12.1 최종 확정 목록

| 구분 | 화면명 | 필수 산출물 | 비고 |
| --- | --- | --- | --- |
| 공통 | 데이터 분포 | `data_distribution.json` | 전체/학습/검증 데이터의 label, feature, cohort 분포 |
| 공통 | Cohort retention | `cohort_summary.json` 또는 `cohort_retention.json` | 모델 전 이탈 현상 설명용 |
| 공통 비교 | 모델별 베이스라인 비교 | `metrics_summary.json` | 최종 3개 모델의 `pr_auc`, `roc_auc` 비교 |
| 모델 상세 | Score distribution | `score_distribution.json` | churn score가 이탈/비이탈을 분리하는지 확인 |
| 모델 상세 | Threshold P/R/F1 | `threshold_curve.json` | threshold별 precision/recall/F1 |
| 모델 상세 | SHAP summary | `shap_summary.json` | 주요 이탈 원인 설명 |
| 비즈니스 | Revenue recovery | `business_value.json` | 회수 가능 매출, 쿠폰/리텐션 시뮬레이션 근거 |

### 12.1.1 data_distribution.json 최소 양식

`data_distribution.json`은 모델별 성능 파일이라기보다 **공통 데이터 설명용 산출물**이다. 모델별로 값이 완전히 다르지 않으면 공통 파일을 하나 만들고 최종 3개 모델 폴더에 복사해도 된다.

권장 양식:

```json
{
  "chart_name": "data_distribution",
  "chart_type": "bar",
  "source": "train_tabular_v2.parquet",
  "label_distribution": [
    {"label": 0, "count": 18329, "ratio": 0.167},
    {"label": 1, "count": 91407, "ratio": 0.833}
  ],
  "split_distribution": [
    {"split": "train", "count": 92971},
    {"split": "test", "count": 109736}
  ],
  "feature_distribution": [
    {"feature": "recency_days", "bin": "0-1", "count": 12000},
    {"feature": "recency_days", "bin": "1-3", "count": 18000}
  ],
  "meta": {
    "schema_version": "artifact.data_distribution.v1",
    "label_name": "churn",
    "horizon_days": 7
  }
}
```

최소 필수:

```text
label_distribution
split_distribution
feature_distribution
```

`feature_distribution`은 모든 피처를 다 넣을 필요는 없고, 발표에서 보여줄 대표 피처 3~5개만 넣어도 된다. 백엔드가 chart-ready로 변환할 때는 `feature`, `bin`, `count`를 우선 사용한다.

### 12.1.2 cohort_summary.json 최소 양식

`cohort_summary.json`은 모델 평가 파일이 아니라 **모델 학습 전 이탈 현상 설명용 공통 산출물**이다. 시계열/코호트 기준으로 잔존율이 어떻게 떨어지는지 보여주는 용도다.

권장 양식:

```json
{
  "chart_name": "cohort_retention",
  "chart_type": "heatmap",
  "source": "canonical_5m",
  "cohorts": [
    {"cohort": "2026-01", "week_index": 0, "users": 10000, "retained_users": 10000, "retention_rate": 1.0},
    {"cohort": "2026-01", "week_index": 1, "users": 10000, "retained_users": 7200, "retention_rate": 0.72},
    {"cohort": "2026-01", "week_index": 2, "users": 10000, "retained_users": 6100, "retention_rate": 0.61}
  ],
  "meta": {
    "schema_version": "artifact.cohort_summary.v1",
    "cohort_unit": "month",
    "period_unit": "week",
    "label_name": "churn"
  }
}
```

최소 필수:

```text
cohort
week_index
users
retained_users
retention_rate
```

대시보드 heatmap은 `cohort`를 y축, `week_index`를 x축, `retention_rate`를 value로 사용한다.

### 12.1.3 서버 처리 기준

현재 FastAPI 백엔드는 `data_distribution`과 `cohort_retention`을 모델별 상세 차트가 아니라 **대시보드 공통 차트**로 본다.

```text
GET /dashboard/charts/data-distribution
GET /dashboard/charts/cohort-retention
```

현재 구현 상태:

| 차트 | 현재 서버 처리 | 비고 |
| --- | --- | --- |
| 데이터 분포 | 서버가 `data/processed/churn/train_tabular_v2.parquet`에서 `recency_days` histogram을 직접 계산 | `data_distribution.json`을 읽는 방식은 아직 아님 |
| Cohort retention | 서버 endpoint는 있으나 현재 `data: []`, `source: pending` 반환 | `cohort_summary.json` reader 추가 필요 |

따라서 지금 기준으로는 `data_distribution.json`과 `cohort_summary.json`의 양식이 없었다는 지적이 맞다. 최종 제출 안정성을 위해 위 양식대로 파일을 만들고, 백엔드는 이후 해당 JSON을 읽도록 보강하는 것이 좋다.

### 12.4 실시간 추천/쿠폰 연동 기준

최종 3개 모델은 추천 알고리즘이 아니라 **향후 7일 이내 이탈 확률 스코어러**다. 실시간 사이트에서는 아래처럼 붙인다.

```text
1. 시뮬레이션 사이트가 view/cart/remove/purchase 이벤트를 보낸다.
2. 백엔드가 이벤트를 10개 실시간 feature로 변환한다.
3. active churn 모델(CatBoost/XGBoost/LightGBM 중 1개)이 churn_probability를 산출한다.
4. churn_probability와 threshold/risk policy로 쿠폰 제공 여부를 결정한다.
5. 현재 이벤트의 category_id/top_category를 seed로 유사 카테고리와 연관 상품을 추천한다.
```

쿠폰 제공은 모델 내부 기능이 아니라 백엔드 정책이다.

```text
churn_probability >= RISK_HIGH  -> 쿠폰/리텐션 메시지 제공
RISK_LOW <= churn_probability < RISK_HIGH -> 추천상품/주의 메시지
churn_probability < RISK_LOW -> 일반 추천만 표시
```

추천 상품은 모델 파일이 아니라 아래 카탈로그 산출물에서 조회한다.

```text
data/processed/recommendation/category_similar.parquet
data/processed/recommendation/category_catalog.parquet
data/processed/recommendation/product_catalog.parquet

또는 시뮬레이션 사이트용:
simulation_site/neon/seed/categories.csv
simulation_site/neon/seed/brands.csv
simulation_site/neon/seed/products.csv
simulation_site/neon/seed/category_similarity.csv
```

### 12.5 서빙 번들 책임 분리

현재 백엔드 실시간 추론 구현은 운영 서빙용 번들인 `models/preprocessors/prep_{Model}_v2.joblib`를 읽을 수 있다. 이 파일들은 모델팀 신규 제출 필수물이 아니라, 기존 전처리/백엔드 쪽에서 생성해 둔 서빙 번들을 운영 경로에 배치해 사용하는 것으로 정리한다.

모델팀 제출 범위:

```text
models/churn/{model_key}/model.*
models/churn/{model_key}/feature_schema.json
models/churn/{model_key}/model_config.json
data/processed/evaluation/churn/{model_key}/model_run_manifest.json
data/processed/evaluation/churn/{model_key}/eval_predictions.parquet
data/processed/evaluation/churn/{model_key}/metrics_summary.json
data/processed/evaluation/churn/{model_key}/score_distribution.json
data/processed/evaluation/churn/{model_key}/threshold_curve.json
data/processed/evaluation/churn/{model_key}/shap_summary.json
data/processed/evaluation/churn/{model_key}/business_value.json
```

백엔드/전처리팀 제공 범위:

```text
models/preprocessors/prep_CatBoost_v2.joblib
models/preprocessors/prep_LightGBM_v2.joblib
models/preprocessors/prep_XGBoost_v2.joblib
simulation_site/neon/seed/categories.csv
simulation_site/neon/seed/brands.csv
simulation_site/neon/seed/products.csv
simulation_site/neon/seed/category_similarity.csv
```

따라서 모델 브랜치 점검 시 `prep_*_v2.joblib`가 없다는 이유만으로 모델팀 산출물 미제출로 보지 않는다. 다만 실시간 API가 실제로 동작하려면 백엔드/전처리팀이 해당 번들을 운영 브랜치에 포함하거나 배포 환경에 배치해야 한다.
